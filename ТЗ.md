Product Brief
1) Короткое описание продукта
Корпоративный сайт-каталог и витрина экспертизы (без онлайн-продаж) для презентации продукции/услуг, кейсов и производственных возможностей, с калькуляторами-вилками и пакетами под ключ. Основная цель — сбор качественных заявок с фиксированием источника и аналитикой. Управление контентом/структурой/SEO/формами — без кода через админку.

2) Цели
Бизнес
    • Рост лидов за счёт “быстрого КП”: вилка цены + сроки + что входит по ключевым категориям.
    • Увеличение доли премиальных сделок
    • SEO-рост не только по категориям/карточкам, но и по интентам: “цена”, “согласование вывески”, “монтаж”, “панель-кронштейн”, “объёмные буквы цена за см”, “неон вывеска цена”.
Пользователь
    • За 1–3 минуты понять: подходит ли решение, сколько стоит примерно, сколько занимает по времени, какие шаги дальше.
    • Быстро увидеть релевантные примеры (кейсы) именно для выбранного типа исполнения.
Операционные
    • Ведение сайта контент-менеджером + быстрый выпуск страниц “под спрос”: прайс-вилки, FAQ по согласованию, страницы “для сетей/агентств”, сервисные страницы.


3) Scope (что входит)
3.1 Публичная часть
Разделы (верхнее меню, по умолчанию)
    • Главная | Продукция | Услуги | Производственные возможности | Проектирование | Портфолио | Контакты
Разделы в футере
    • Дублирование основных разделов + Новости + юридические документы. Новости — только в футере, но полноценный SEO-раздел (листинг + карточка новости).
Каталог
    • Вложенные URL:
        ◦ Категория: /produkciya/{category_slug}
        ◦ Товар: /produkciya/{category_slug}/{product_slug}
    • Категории — SEO-страницы с описанием и листингом.
    • Каждый “тип исполнения” — отдельная карточка товара.
    • Характеристики товара: плоский список key→value.
    • На карточке товара — блок «Примеры работ» из портфолио (связанные кейсы).
Услуги
    • 11 услуг, каждая — отдельная страница с контентом+SEO, формами и (опционально) кейсами. 
Портфолио
    • Листинг + карточка кейса.
    • Фильтры: по товарам/услугам + поиск.
    • NDA: возможность скрыть клиента на публичной части.
    • На странице кейса: ссылки на связанные товары и услуги.
Поиск
    • Единое поле по сайту:
        ◦ autocomplete в шапке
        ◦ страница результатов /search?q=... с группировкой по типам контента.
Формы
    • Типы: звонок / расчет / вопрос.
    • Обязательные поля во всех формах: телефон + email + чекбокс согласия на обработку ПДн.
    • Фиксация источника: URL, название страницы, UTM.
Контакты
    • Одна точка, карта, реквизиты, телефон, почта, мессенджеры: Telegram + WhatsApp.

3.2 Админ-панель (Laravel + Filament/Orchid)
Контент без кода
    • CRUD: категории/товары/услуги/портфолио/новости/статические страницы.
    • Page Builder (секции/блоки) для главной и статических страниц.
    • Глобальные (переиспользуемые) блоки: формы/CTA, контакты и др.
Заявки
    • Список заявок, статусы: Новая/В работе/Закрыта, одно поле комментария, экспорт CSV.
    • Уникальность для аналитики: «уникальная заявка по контактам».
Импорт
    • CSV-импорт: товары, услуги, портфолио, заявки.
    • Медиа после импорта — вручную.
SEO
    • SEO-шаблоны с переменными + override на сущностях.
    • Микроразметка Schema.org (шаблонно + override).
Аналитика
    • Интеграции: Яндекс Метрика + Google Analytics.
    • События: отправка форм, клики по телефону, клики по мессенджерам (Telegram/WhatsApp).
Аудит
    • Логирование действий пользователей админки (кто/что/когда).

4) Non-Goals (что не входит)
    • Корзина/оплата/доставка, личный кабинет клиента.
    • Многогородность.
    • Обязательная 2FA и IP-ограничения.
    • Bulk-операции по товарам (массово скрыть/перенести и т.п.).

5) Стартовые объемы (для проектирования и тестов)
    • Категорий: ~10
    • Товаров: ~100 (в среднем 10 на категорию)
    • Кейсов: ~15 на старте, далее пополнение по мере работ
    • Новостей: 3 на старте, далее еженедельно (публикации через админку)

6) Нефункциональные требования (минимум)
    • Производительность: «быстро на типовом VPS» (оптимизация изображений, кеширование, предсказуемая админка).
    • Безопасность: HTTPS, rate limit на логин админки, антиспам на формах при необходимости, бэкапы БД/медиа, ротация логов.

7) Метрики успеха
    • Воронка: уникальные посетители → уникальные заявки (по контактам).
    • Конверсии: отправки форм; клики tel:/Telegram/WhatsApp.
    • Контент/SEO: индексация и рост органики на категории/товары/услуги/новости.

Wireframes (текстовые)
0) Глобальные правила для всех страниц
0.1 Хедер (фиксированный/обычный — решается в UI, но структура одна)
    1. Логотип (ссылка на /)
    2. Верхнее меню: Главная / Продукция / Услуги / Производственные возможности / Проектирование / Портфолио / Контакты
    3. Поиск (поле) + иконка
    4. Контакты-CTA: телефон (tel:), кнопка Telegram, кнопка WhatsApp (клики считаются событиями)
    5. Кнопка “Оставить заявку” (открывает выбор типа формы или сразу “Расчет” — фиксируем как стандарт)
0.2 Autocomplete (поиск в хедере)
    • При фокусе: выпадающий слой
        ◦ строка ввода + крестик очистки
        ◦ блоки результатов (до N):
            ▪ Товары
            ▪ Услуги
            ▪ Портфолио
            ▪ Новости
            ▪ Страницы (опционально)
        ◦ “Показать все результаты” → /search?q=...
0.3 Футер
    1. Меню: (дубли верхнего) + Новости
    2. Документы: Политика ПДн / Согласие / Пользовательское соглашение / Cookies
    3. Контакты: телефон, email, Telegram, WhatsApp
    4. Реквизиты (кратко) (если требуется)
    5. Копирайт
0.4 Общие компоненты
    • Breadcrumbs на всех страницах кроме главной (и, опционально, хабов).
    • Заголовок H1 на каждой странице (кроме главной — по дизайну).
    • CTA/Формы: могут быть
        ◦ встроенным блоком на странице
        ◦ или открываться модальным окном
        ◦ в обоих случаях логика одинаковая.
0.5 Формы (единый шаблон)
    • Заголовок (по типу): “Заказать звонок” / “Получить расчет” / “Задать вопрос”
    • Поля (обязательные): Телефон + Email
    • Поле “Комментарий/Сообщение” (опционально, в зависимости от типа формы; по умолчанию есть в “Вопрос” и “Расчет”)
    • Чекбокс (обязательный): “Согласен на обработку персональных данных” + ссылка на /privacy и /consent
    • Кнопка отправки
    • Сообщения состояний: успех / ошибка / антиспам-блокировка
    • Скрытые данные: URL страницы, название страницы, UTM

1) Главная /
Цель: быстро показать направления, доверие (кейсы), CTA на заявку.
Desktop-порядок блоков
    1. Hero:
        ◦ Заголовок + подзаголовок
        ◦ 1–2 CTA-кнопки: “Получить расчет” (primary), “Связаться” (secondary)
        ◦ мини-контакты (телефон/мессенджеры)
    2. Блок “Продукция” (плитка категорий 10 шт., 2–4 в ряд)
    3. Блок “Услуги” (список/иконки, 11 шт.)
    4. Блок “Примеры работ” (превью кейсов 6–9 + “Все кейсы” → /portfolio)
    5. Блок “Производственные возможности” (коротко: 3–6 карточек-сущностей, без ссылок, + кнопка “Подробнее” → /proizvodstvennye-vozmozhnosti)
    6. Блок преимуществ / этапов (опционально)
    7. CTA-форма (глобальный блок) “Получить расчет”
    8. Контакты (краткий блок с картой-превью и ссылкой на /kontakty)
Mobile
    • Все в одну колонку; плитки в 1–2 колонки, кейсы карусель/лента.

2) Хаб “Продукция” /produkciya
Цель: выбор категории.
    1. H1 “Продукция”
    2. Базовый текст (короткий абзац)
    3. Сетка категорий:
        ◦ карточка: изображение/иконка + название + количество товаров (опционально)
    4. CTA-блок (опционально): “Нужна консультация/расчет?”

3) Категория продукции /produkciya/{category_slug}
Цель: SEO-лендинг + выбор товара + CTA.
    1. Breadcrumbs: Главная → Продукция → {Категория}
    2. H1 Категории
    3. Вводный блок:
        ◦ краткое описание (SEO-текст сверху)
        ◦ баннер/изображение (опционально)
    4. Листинг товаров (карточки):
        ◦ фото, название, 1–2 коротких факта (опционально), кнопка “Подробнее”
        ◦ сортировка (опционально): “По умолчанию / Новые”
    5. Блок “Примеры работ” (кейсы по товарам этой категории):
        ◦ 6 превью, “Все кейсы” с предфильтром → /portfolio?category=... (или через товары категории)
    6. CTA-блок (встроенный или кнопка открытия формы):
        ◦ primary: “Получить расчет”
        ◦ secondary: “Задать вопрос”

4) Карточка товара /produkciya/{category_slug}/{product_slug}
Цель: показать товар + убедить + привести к заявке, подкрепить кейсами.
    1. Breadcrumbs: Главная → Продукция → Категория → Товар
    2. H1 Название товара
    3. Верхний блок “галерея + краткое описание”:
        ◦ Галерея (слайдер, превью)
        ◦ справа: короткое описание/ключевые тезисы
        ◦ быстрые CTA: “Расчет”, “Вопрос”, “Звонок”
    4. Блок “Описание” (текст)
    5. Блок “Характеристики” (таблица key→value, плоский список)
    6. Блок “Примеры работ” (кейсы, связанные с товаром):
        ◦ превью 6–9
        ◦ фильтр внутри блока (опционально): “Все / свежие”
        ◦ “Все кейсы по этому товару” → /portfolio?product=...
    7. CTA-форма (встроенная) или повтор CTA-кнопок
    8. SEO/микроразметка: выводится “под капотом” (не UI-блок)

5) Услуги — листинг /uslugi
Цель: выбрать услугу.
    1. Breadcrumbs (опционально)
    2. H1 “Услуги”
    3. Список услуг (карточки/строки):
        ◦ название, 1 строка описания (опционально), “Подробнее”
    4. CTA-блок “Получить расчет” (опционально)

6) Карточка услуги /uslugi/{service_slug}
Цель: объяснить услугу + привести к заявке.
    1. Breadcrumbs: Главная → Услуги → {Услуга}
    2. H1 услуги
    3. Контентные блоки (page builder или структурные секции):
        ◦ описание
        ◦ этапы/преимущества (опционально)
    4. CTA-блок: “Получить расчет” / “Задать вопрос”
    5. Блок “Примеры работ” (опционально, если включено):
        ◦ кейсы, связанные с услугой
    6. CTA-форма (опционально)

7) Производственные возможности /proizvodstvennye-vozmozhnosti
Цель: доверие и демонстрация мощностей.

    3. Набор карточек (некликабельные сущности):
        ◦ фото/иконка
        ◦ название
        ◦ краткое описание

    4. CTA-блок “Получить расчет/Консультацию”

8) Проектирование /proektirovanie
Цель: объяснить процесс проектирования + заявка.
    1. Breadcrumbs

9) Портфолио — листинг /portfolio
Фиксация: только листинг/фильтры/поиск.
    1. Breadcrumbs
    2. H1 “Портфолио”
    3. Панель фильтров:
        ◦ фильтр “Товары” (мультивыбор или single-select)

    4. Сетка кейсов:
        ◦ превью-фото

Цель: показать результат + вести к товарам/услугам + CTA.

    2. H1 Название кейса

    5. Описание кейса
    6. Блок “Связанные товары” (список ссылок/карточек)
    7. Блок “Связанные услуги” (список ссылок/карточек)
    8. CTA-блок: “Получить расчет по похожему проекту” (форма)

Функциональное ТЗ
1) Рамки и принципы
    • Тип: интернет-каталог + услуги + портфолио + новости + документы, без продаж/корзины/оплат/ЛК.
    • Технологии: Laravel; админка Filament или Orchid; файлы — локально на сервере.
    • Цель: сбор заявок (лидов) и удобное самостоятельное ведение контента через админку.
    • Контент-процесс: черновик/публикация + предпросмотр, публиковать может контент-менеджер.
    • Стартовые объемы для проектирования/тестов: ~10 категорий, ~100 товаров, ~15 кейсов, 3 новости на старте, далее новости еженедельно.

2) Роли и доступы (RBAC)
Роли: Супер-админ / Админ / Контент-менеджер.
2.1 Матрица прав (обязательная)
    • Супер-админ
        ◦ Все права + системные настройки (интеграции аналитики, резервные копии/настройки хранения, SEO-шаблоны, роли/пользователи).
    • Админ
        ◦ Управление контентом, формами/заявками, статистикой; без системных настроек уровня инфраструктуры (по необходимости — уточняется внутри реализации).
    • Контент-менеджер
        ◦ CRUD контента (категории/товары/услуги/портфолио/новости/страницы), публикация, загрузка медиа.
        ◦ Без доступа к системным настройкам (например SEO-шаблоны, аудит/экспорт логов, интеграции — по решению заказчика).

10) Критерии приемки (минимальные, функциональные)
    • Любой контентный объект можно создать/редактировать/опубликовать/снять с публикации без кода.
    • Все формы: обязательные поля + согласие ПДн + фиксация источника + уведомления Email/Telegram + сохранение заявки в админке.
    • На товарах/услугах отображаются связанные кейсы; на кейсах — ссылки на товары/услуги.
    • Поиск работает: autocomplete + страница результатов с группировкой.
    • Новости индексируются и имеют листинг/карточку (при этом пункт доступен из футера).
    • Логи админки фиксируют действия; есть rate limit на логин; бэкапы и ротация логов предусмотрены.

5) Критические правила на уровне API/БД
    • Смена slug у категорий/товаров/кейсов/новостей/страниц:
        ◦ создавать redirects (301) со старого пути на новый.
    • Публикация:
        ◦ публичные public endpoints возвращают только status=published.
    • NDA:
        ◦ в public API для кейса при is_nda=true скрывать client_name, отдавать public_client_label или “NDA” по настройке.
    • Уникальность заявок:
        ◦ дубль не блокирует создание leads, влияет только на расчет “уникальных” через lead_dedup_index.

    • Хлебные крошки: обязательны на всех страницах ниже 1 уровня (категория, товар, услуга, кейс, новость, документы).
    • Липкий хедер: допускается, но не обязателен; важнее, чтобы поиск и контакты были всегда “под рукой” на mobile.

3) Адаптивность и сетка
    • Поддержка: desktop + mobile + корректная работа на промежуточных ширинах.
    • Рекомендованные брейкпоинты (не догма): 360–430 (mobile), 768 (tablet), 1024 (small desktop), 1280+ (desktop).
    • Сетка: 12 колонок на desktop; 4–6 на mobile/tablet.
    • Контент не должен “прыгать” при загрузке изображений (предусмотреть фиксирование пропорций/плейсхолдеры).

4) Визуальный стиль и дизайн-система
    • Стиль: современный, “производственный”, без визуального шума; акцент на фото работ и продукцию.
    • Единые компоненты (атомарная система):
        ◦ Кнопки: primary / secondary / text
        ◦ Поля ввода: стандартные состояния (default/focus/error/disabled)
        ◦ Карточки: товар/категория/кейс/новость
        ◦ Бейджи: “NDA”, “Новое” (опционально), статусы
        ◦ Алёрты: успех/ошибка/инфо
        ◦ Пагинация или “Показать ещё” (единообразно внутри сайта)
    • Типографика: 1–2 шрифта, чёткая иерархия H1–H3, читабельность на mobile.
    • Иконки: единый набор (один стиль), особенно для мессенджеров и преимуществ.

5) Контентные принципы
    • Фото/портфолио — ключевой носитель доверия:
        ◦ В карточке товара/услуги кейсы размещаются выше первого “глубокого” CTA либо сразу после характеристик/описания.
    • Тексты категорий: короткий верхний текст + при необходимости расширяемый блок (но базово — без перегруза).
    • Даты: локаль RU, формат единый (например: DD.MM.YYYY или Месяц ГГГГ) — выбрать и использовать везде.

6) Формы заявок (ключевой UX-блок)
Типы: звонок / расчет / вопрос.
Обязательные элементы UI:
    • Поля: телефон + email (обязательные)
    • Чекбокс согласия на обработку ПДн (обязательный) + ссылки на документы
    • Понятный заголовок формы (по типу заявки)
    • Состояния отправки:
        ◦ loading (блокировка повторной отправки)
        ◦ success (подтверждение и что делать дальше)
        ◦ error (сообщение + подсветка полей)
    • Антиспам: не ухудшать UX (в идеале невидимые механизмы; явная капча только при необходимости).
UX-требования:
    • Телефон: маска и понятные подсказки, допускается автоподстановка +7.
    • Валидации: показывать ошибки сразу после попытки отправки и/или после выхода из поля (не “кричать” заранее).
    • На mobile: удобная клавиатура для телефона/email.
    • Форма должна явно “объяснять” согласие: текст + ссылки.

7) Поиск (autocomplete + страница результатов)
Autocomplete (в хедере):
    • Появляется при вводе.
    • Группирует результаты: товары/услуги/портфолио/новости (страницы опционально).
    • Ограничение результатов на группу (например 3–5), кнопка “Все результаты”.
/search?q=…
    • Группировка (вкладки или блоки).
    • Пустые состояния: “ничего не найдено” + подсказки (проверить запрос, перейти в продукцию).
    • На mobile результаты должны читаться “сканированием”: заголовок + тип + короткий сниппет.

8) Каталог и карточки (UX-паттерны)
Хаб “Продукция” (/produkciya):
    • Сетка категорий (крупные кликабельные карточки).
    • Базовый текст короткий, не мешает выбору.
Категория (/produkciya/{category}):
    • Вводный блок (H1 + краткий текст).
    • Листинг товаров:
        ◦ карточка товара: фото + название + краткий тезис (опционально) + явная кликабельность.
    • Встроенный CTA-блок внизу/середине (но не “давить” сверху).
    • Блок кейсов по категории (если выводится) — визуально отличать от товаров.
Товар (/produkciya/{category}/{product}):
    • “Первый экран” = галерея + краткая суть + CTA.
    • Характеристики key→value:
        ◦ читабельная таблица, одинаковое выравнивание, перенос длинных значений.
    • Кейсы (“Примеры работ”) — с хорошими превью, открытие кейса в 1 клик.

9) Портфолио
/portfolio (листинг):
    • Фильтры по товарам и услугам + поиск.
    • Фильтры должны быть удобны на mobile (селекты/чипсы, не перегружать).
    • Карточка кейса:
        ◦ фото, название, дата
        ◦ клиент: скрывать при NDA (показывать “NDA” или скрывать строку — по настройке).
Кейс (/portfolio/{case}):
    • Галерея — центральный элемент (поддержка большого количества фото).
    • Блок “Связанные товары” и “Связанные услуги” — визуально как ссылки/карточки для перехода обратно в коммерческие страницы.
    • CTA: “Получить расчет по похожему проекту” (не обязательно всегда, но рекомендуется как глобальный/локальный блок).

10) Контакты и конверсионные клики
    • Телефон, Telegram, WhatsApp должны быть заметны:
        ◦ в хедере и на странице контактов
        ◦ кликабельность очевидна (кнопки/иконки + подписи).
    • На контактах:
        ◦ карта не должна “ломать” скролл на mobile (желательно статичный превью + кнопка “Открыть карту” или аккуратный embed).

11) Доступность и UX-качество
    • Контраст текста/кнопок: достаточный для чтения (особенно на mobile и на фото-фонах).
    • Состояния фокуса (keyboard) — обязательно (включая поиск, фильтры, формы).
    • Alt-тексты для изображений:
        ◦ обязательны минимум для ключевых (обложки категорий/товаров/кейсов), остальное — опционально.
    • Понятные пустые состояния:
        ◦ нет товаров в категории (на этапе наполнения)
        ◦ нет кейсов по фильтру
        ◦ нет новостей
        ◦ ошибка загрузки

12) Требования к медиа (UI + производительность)
    • Изображения:
        ◦ одинаковые пропорции превью в листингах (чтобы сетка была ровной)
        ◦ оптимизация (WebP/AVIF при возможности, lazy-load вне первого экрана)
        ◦ для галерей: поддержка увеличения/просмотра
    • Видео: не является обязательным функционалом (если появится — отдельное ТЗ).

13) Требования к page builder (влияющие на UX)
    • Единый набор секций, визуально согласованный.
    • Секции должны иметь:
        ◦ заголовок (опционально)
        ◦ фон/отступы/варианты сетки (ограниченный набор)
        ◦ порядок/включение-выключение
    • Глобальные блоки (CTA, контакты) должны выглядеть одинаково на всех страницах.

14) UX критерии приемки (Definition-ready)
    • Пользователь за ≤3 клика может:
        ◦ из главной попасть в категорию и в товар;
        ◦ из товара перейти в кейс и обратно в товары/услуги.
    • Любая форма:
        ◦ имеет обязательные поля + согласие,
        ◦ показывает понятные ошибки,
        ◦ показывает подтверждение успешной отправки.
    • Поиск:
        ◦ autocomplete показывает результаты по группам,
        ◦ /search показывает результаты по группам и корректные пустые состояния.
    • NDA в кейсе:
        ◦ клиент скрывается на публичной части согласно настройке.
    • Все ключевые элементы корректно отображаются на mobile (особенно фильтры портфолио и формы).

Definition of Done (DoD) v2.2
0) Общие условия приемки

    • Все функции доступны и проверяемы без доступа к исходникам (через UI сайта/админки и API при необходимости).

        ◦ ~10 категорий, ~100 товаров, ~15 кейсов, 3 новости, страницы и документы.

2) Функциональные критерии — публичная часть
2.1 Навигация, страницы, URL

    • Футер содержит ссылки: основные разделы + Новости + документы.
    • Каталог имеет вложенные URL:
        ◦ /produkciya/{category_slug}
        ◦ /produkciya/{category_slug}/{product_slug}
    • Новости доступны из футера, но работают как полноценный SEO-раздел: /news, /news/{news_slug}.
    • Все страницы корректно отдают 404 для несуществующих URL, 500 не возникает при штатных действиях.
2.2 Каталог
    • /produkciya показывает сетку категорий + базовый текст.
    • Страница категории содержит: H1, описание, листинг товаров, CTA-блок(и).

    • Перелинковка работает:
        ◦ товар → кейсы
        ◦ кейс → связанные товары/услуги
2.3 Услуги
    • /uslugi содержит листинг 11 услуг.
    • Страница услуги: контент + SEO + CTA/формы + (если включено) кейсы.
2.4 Портфолио
    • /portfolio — только листинг + фильтры (по товарам/услугам) + поиск, без дополнительных секций.
    • Карточка кейса содержит: галерею, описание, дату, клиент (с NDA-логикой), связанные товары/услуги.
    • NDA работает: при включенном NDA клиент скрыт публично (показывается “NDA” или скрывается строка — согласно настройке реализации).
2.5 Контакты
    • /kontakty содержит: карту, реквизиты, телефон, email, Telegram, WhatsApp.
    • Клики по телефону/мессенджерам возможны и считаются событиями аналитики.
2.6 Поиск
    • Autocomplete в хедере: подсказки с группировкой (товары/услуги/кейсы/новости).
    • /search?q=...: результаты сгруппированы, есть “ничего не найдено”.

3) Формы и заявки — критерии готовности
3.1 Формы (публичная часть)
    • Доступны 3 типа: звонок/расчет/вопрос.
    • Обязательные поля всегда: телефон + email + чекбокс согласия ПДн с ссылками на документы.
    • Фиксируется источник: URL страницы + (если доступно) название страницы + UTM.
    • Есть состояния: отправка/успех/ошибка, понятные сообщения пользователю.
3.2 Заявки (админка)
    • Каждая отправка формы создает заявку в админке.
    • Заявка включает: тип, поля, источник, UTM, согласие (факт и время), статус, комментарий.
    • Статусы: Новая / В работе / Закрыта (смена статуса сохраняется).
    • Комментарий: одно текстовое поле, сохраняется.
3.3 Уведомления
    • Email-уведомления работают (на заданных получателей).
    • Telegram-уведомления работают (на заданные чаты/получателей).
    • Ошибка доставки уведомления не мешает сохранению заявки (фиксируется в логах).
3.4 Экспорт CSV
    • Экспорт заявок в CSV работает с фильтрами (период/статус/тип/источник/UTM).
    • CSV корректно открывается в Excel/LibreOffice (UTF-8, корректные разделители/экранирование).

4) Админ-панель — критерии готовности
4.1 Контент и публикации
Для категорий/товаров/услуг/кейсов/новостей/страниц/документов:
    • CRUD работает.
    • Черновик/публикация работает.
    • Предпросмотр работает.
    • SEO-поля доступны и применяются на публичной части.
    • Slug редактируемый; при изменении slug создается 301-редирект.
4.2 Связи
    • Кейсы можно связать с несколькими товарами и услугами.
    • На товаре/услуге отображаются связанные кейсы.
    • На кейсе отображаются связанные товары/услуги.
4.3 Page Builder и глобальные блоки
    • Главная и статические страницы редактируются секциями.
    • Глобальные блоки создаются и вставляются на страницы.
    • Изменение глобального блока отражается на всех страницах, где он используется.
4.4 Меню
    • Верхнее и нижнее меню настраиваются: порядок, видимость, ссылки.
4.5 Импорт CSV
    • Импорт работает для: товары, услуги, портфолио, заявки.
    • Медиа после импорта добавляется вручную, это не блокирует публикацию.
    • Импорт выдает отчет об ошибках (строки/причины).
4.6 Аудит админки
    • Логирование действий админ-пользователей включено:
        ◦ кто/что/когда
        ◦ минимум: create/update/delete/publish/unpublish/login
    • Логи доступны супер-админу (и админу — если разрешено в настройках ролей).

5) SEO и индексация — критерии готовности
    • Генерация meta title/description/h1 по шаблонам с переменными работает (где задано).
    • Override SEO на сущности перекрывает шаблон.
    • /sitemap.xml содержит индексируемые страницы (включая новости и документы).
    • /robots.txt корректен.
    • Schema.org выводится на релевантных типах страниц (как минимум товар/услуга), без ошибок разметки.

6) Аналитика — критерии готовности
    • Метрика и GA подключены и включены на всех страницах.
    • События фиксируются:
        ◦ отправка форм
        ◦ клики tel:
        ◦ клики Telegram
        ◦ клики WhatsApp
    • В админке отображается:
        ◦ воронка “уникальные посетители → уникальные заявки”
        ◦ уникальные заявки считаются по контактам (дедупликация для отчетов).

7) Производительность, надежность, безопасность — критерии готовности
    • Сайт “быстро работает на типовом VPS” при стартовых объемах контента:
        ◦ страницы открываются без ощутимых задержек,
        ◦ листинги не “падают” и не таймаутят.
    • Включен rate limit на логин админки.
    • Формы защищены от базового спама (механизм выбран в реализации).
    • HTTPS включен.
    • Настроены бэкапы БД и медиа (описана политика и шаги восстановления).
    • Логи приложения ротируются.

8) Качество и тестирование — критерии готовности
8.1 Минимальный набор тестов
    • Smoke тесты (ручные) по чек-листу: навигация, формы, заявки, публикации, поиск.
    • Проверка адаптива (минимум: iPhone/Android mobile + desktop).
    • Проверка импорта CSV (как минимум по 1 файлу на каждую сущность).
    • Проверка редиректов при смене slug (категория/товар/новость/страница).
    • Проверка NDA для кейса.
8.2 Баг-классификация (приемка)
    • Blocker: форма не отправляется/заявки не сохраняются/админка недоступна/падения — не допускается.
    • Major: критично нарушает основные сценарии (каталог/поиск/публикации/редиректы) — не допускается.
    • Minor: визуальные/текстовые/единичные UI дефекты — допускается только при наличии списка и сроков исправления.
    • Trivial: косметика — допускается.

9) Артефакты сдачи (что должно быть передано)
    • Доступы: production/staging, админ-пользователи.
    • Инструкция:
        ◦ как добавлять/публиковать контент,

        ◦ как настраивать формы/получателей,
        ◦ как делать импорт CSV,
        ◦ как смотреть заявки/экспортировать CSV.
    • Инструкция эксплуатации:
        ◦ резервное копирование и восстановление (БД + медиа),
        ◦ ротация логов,
        ◦ обновление зависимостей по регламенту.
    • Список интеграций: ID счетчиков Метрики/GA, Telegram-каналы/чаты уведомлений.
    • Финальный набор документов проекта (в актуальных версиях).
